## 1. 專案概述 (Project Overview)

- **專案背景與目標**：Multi-Lang Coach 是一款由大型語言模型 (LLM) 驅動的互動式語言學習平台。解決傳統語言學習缺乏「真實情境演練」與「即時專業糾錯」的痛點。
- **目標受眾**：希望在職場 (如 IT、HR、行銷) 或日常生活情境中，精進英語或日語口說與對話流暢度、準確度的使用者。
- **核心功能亮點**：
  - 基於 Google Gemini API 的高擬真情境對話。
  - 即時語音辨識 (支援跨平台 Web Speech API 與 Gemini 語音直接辨識)。
  - 即時文法糾錯與單字/片語萃取。
  - PDF 學習報告匯出功能。
  - 單字本 (Notebook) 與句型庫 (Pattern Notebook) 的個人化整理。

## 2. 系統架構與技術選型 (System Architecture & Tech Stack)

- **前端框架**：React 18 + Vite (純前端架構，無後端伺服器)。
- **路由機制**：無傳統 Router，透過 `App.jsx` 的 State (`activeTab`) 進行元件切換。
- **UI/UX 樣式**：Vanilla CSS (`App.css`, `index.css`)，全面採用 CSS Variables 實作深色主題與玻璃擬物化 (Glassmorphism) 設計。
- **資料持久化**：使用瀏覽器的 `localStorage` 儲存所有資料。
- **第三方整合**：Google Gemini API (`v1beta`)、Web Speech API、`html2pdf.js`。

## 3. 功能模組與元件規劃 (Module Specifications)

專案組件集中於 `src/components`，主要劃分如下：

### 3.1 App.jsx (核心容器與全域狀態)
作為整個應用的 Entry Point，負責管理所有的全域狀態（API Key、Target Language、User Context 等），並作為 Props 傳遞給子元件。同時負責渲染 Settings (設定介面) 與 Welcome Modal。

### 3.2 Chat.jsx (對話模組 - 系統核心)
處理與 LLM 的對話呈現、語音輸入邏輯、TTS 朗讀。
> [!NOTE]
> 支援「混合模式語音輸入」：電腦端預設使用原生 Web Speech API。Android 端則啟動特殊機制（詳見第 9 章邊界條件），改採音檔錄製轉 Base64 後由 Gemini API 解析。

## 4. 資料模型與狀態管理 (Data Models)

應用程式依賴 `localStorage` 來確保使用者關閉網頁後資料不遺失。我們使用 TypeScript Interface 來嚴謹定義核心資料模型：

### 4.1 Chat History Schema (對話紀錄模型)
```typescript
interface ChatMessage {
  role: 'system' | 'user' | 'assistant';
  content: string;               // 訊息內文
  translation?: string;          // 繁體中文翻譯
  correction?: {                 // 文法糾錯 (若無錯誤則為 null)
    original: string;
    error: string;
    fixed: string;
    explanation: string;
  };
  extractedVocab?: {             // 從對話中擷取的新單字
    term: string;
    phonetic: string;
    partOfSpeech: string;
    meaning: string;
    example: string;
  };
}
```

### 4.2 LocalStorage 鍵值映射 (Keys)
```typescript
interface LocalStorageKeys {
  APP_GEMINI_API_KEY: string;         // API 憑證
  APP_USER_CAT: string;               // 使用者分類 (如: business)
  APP_USER_ROLE: string;              // 角色 (如: it)
  APP_USER_LEVEL: string;             // 程度 (如: pre-intermediate)
  IT_APP_TARGET_LANG: 'en' | 'ja';    // 學習目標語言
  IT_ENGLISH_APP_VOCABULARY: Array;   // 單字本資料
  APP_LEARNING_PROGRESS: Object;      // 連續登入與完成任務數
}
```

## 5. AI 引擎與 Prompt Engineering

底層統一呼叫 `src/utils/llmClient.js`。目前系統全面升級並鎖定使用：**`gemini-2.5-flash`**。

- **`callGeminiAPI` (情境對話)**：根據使用者的 Level 給予角色設定。嚴格要求模型回傳符合 `ChatMessage` 介面的純 JSON 格式。
- **`analyzeSentenceAPI` (句子深度解析)**：針對單一外語句子，回傳單字、句型與文法解析。
- **`polishSentenceAPI` (潤飾與翻譯)**：將使用者的破碎草稿，配合前文對話，翻譯成完美的外語。
- **`transcribeAudioWithGemini` (高精度語音辨識)**：傳入 Base64 音訊，指示模型「只回覆聽寫出的文字，絕對不要加上任何解釋」。

## 6. 靜態資料庫設計 (Static Data)

位於 `src/data`，作為無後端架構的資料支撐：
- **`categoryData.js`**：定義可選的 Categories、Roles 與 Levels。
- **`scenariosData.js`**：提供 `getScenariosByRole` 函數，依據角色動態生成 6 個情境任務 + 1 個自由對話。
- **`scenarioPatterns_01/02.json`**：預先整理好的海量句型庫，以情境 ID 為 Key。

---

## 7. 未來擴展與維護指南 (Maintenance)

- **新增情境或角色**：修改 `categoryData.js` 新增 Role ID，並於 `scenariosData.js` 加入情境邏輯。
- **替換底層 LLM (例如 OpenAI)**：僅需重寫 `llmClient.js` 內部的 `fetch` 邏輯與 Payload 格式，保持對外的 export functions (如 `callGeminiAPI`) 輸入輸出一致即可，前端 UI 零修改。

## 8. AI 代理重構指令碼 (AI Agent Reconstruction Directive)

*This section is strictly formatted for LLM/AI Agent parsing. Do NOT modify unless updating core architecture constraints.*

### [Repository Skeleton]
```text
/src
 ├─ App.jsx (Global State Hub)
 ├─ main.jsx (React Root)
 ├─ App.css & index.css (CSS variables & Glassmorphism classes)
 ├─ /components (Chat, Dashboard, Notebook, PatternNotebook, Patterns, Guide, Sidebar)
 ├─ /utils (llmClient.js, pdfExporter.js)
 ├─ /data (categoryData.js, scenariosData.js, scenarioPatterns_*.json)
 └─ /contexts (I18nContext.jsx)
```

### [State Flow Topology]
```mermaid
flowchart LR
    State((App.jsx<br>State)) --"apiKey, targetLanguage,<br>userCategory, userRole"--> Chat.jsx
    State --"vocabulary,<br>savedPatterns"--> Notebook.jsx
    State --"activeTab"--> View[Rendered Component]
```
All states in `App.jsx` MUST have an equivalent `useEffect` that serializes and saves to `localStorage` on change.

### [Core Logic Constraints (CRITICAL)]

> [!CAUTION]
> **1. LLM JSON Fallback (CRITICAL)**: Inside `llmClient.js`, when parsing LLM responses for JSON, you MUST implement a fallback regex to strip ` ```json ` and ` ``` ` tags. Gemini frequently wraps JSON in markdown blocks despite explicit system instructions.

> [!WARNING]
> **2. Android STT Bug Workaround**: Android Chrome Web Speech API duplicates text continuously. Inside `Chat.jsx` `toggleRecording`, you MUST check:
> `const shouldUseGeminiSTT = /Android/i.test(navigator.userAgent) && androidSmartSpeech && apiKey;`
> If true, bypass native SpeechRecognition completely. Use `MediaRecorder` -> `blobToBase64` -> `transcribeAudioWithGemini` to ensure text accuracy.

> [!IMPORTANT]
> **3. TTS Multi-Lang Segmentation**: In `Chat.jsx` `handleSpeak`, English sentences containing Chinese translations MUST be split by regex (CJK characters) into separate `SpeechSynthesisUtterance` instances. Do not rely on the browser to auto-switch languages mid-sentence.

> [!IMPORTANT]
> **4. Model String Enforcement**: ALL API fetch calls in `llmClient.js` must construct the URL using exactly `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=${apiKey}`.

### [Sequential Build Steps]
1. **Init**: Create Vite React app. Build `index.css` with CSS variables.
2. **Data**: Define `src/data` static structures.
3. **Core Utils**: Build `llmClient.js` with strictly formatted API payloads.
4. **UI**: Build purely functional side components (`Sidebar`, `Dashboard`).
5. **Chat Engine**: Build `Chat.jsx`. Implement STT MediaRecorder fallback, TTS segmentation, and LLM handlers.
6. **Assembly**: Wire everything into `App.jsx` with full `localStorage` syncing.
